Redis 基础数据结构及特性

发表于 2018-09-25 | 分类于 Redis

Redis 被广泛应用的互联网存储中间件。
Redis 超高的性能、完美的文档、简洁易懂的源码以及丰富的客户端库（Java、PHP、Python、Go、Ruby 等）。
Redis 互联网后端工程师的必备技能之一。
Redis 一个开源的（BSD 协议），基于内存的高性能非关系型（NoSQL）数据库。
Redis 常被用作数据库、缓存、消息中间件等。
Redis 的特性：

  • 支持数据的持久化
  • 数据类型丰富：STRING、HASH、LIST、SET、ZSET
  • 单线程
  • 原子化操作
  • 分布式

Redis 所有的数据结构都是以唯一的 KEY 作为名称的，也只能通过 KEY 来获取存储的数据。不同类型的数据结构 VALUE 的结构不一样。

结构类型 结构存储的值 结构的读写能力 特性
STRING 可以是字符串、整数或者浮点数 对整个字符串或者字符串的其中一部分执行操作；对整数和浮点数执行自增（increment）或者自减（decrement）操作 动态更新，预分配冗余空间
LIST 一个链表，链表上的每个节点都包含了一个字符串 从链表的两端推入或者弹出元素；根据偏移量对链表进行修剪（trim）；读取单个或者多个元素；根据值查找或者移除元素 有序、重复、自动创建、自动回收
HASH 包含键值对的无序散列表 添加、获取、移除单个键值对；获取所有键值对 无序、自动创建、自动回收
SET 包含字符串的无序收集器（unordered collection），并且被包含的每个字符串都是独一无二、各不相同的 添加、获取、移除单个元素；检查一个元素是否存在于集合中；计算交集、并集、差集；从集合里面随机获取元素 无序、唯一、自动创建、自动回收
ZSET 字符串成员（member）和浮点数分值（score）之间的有序映射，元素的排列顺序由分值大小决定 添加、获取、移除单个元素；根据分值范围（range）或者成员来获取元素 有序、唯一、自动创建、自动回收

MYSQL 表结构设计最佳实践

发表于 2017-08-04 | 更新于 2018-09-06 | 分类于 MYSQL

命名规则

  1. 表名、字段名只能使用小写字母、数字、下划线，尽量采用语义明确且合适的单词
  2. 多个单词使用下划线下划线链接，且不超过 3 个，总长度不超过 30 个字符
  3. 禁止使用 SQL 关键字，如 desc，range，match 等
  4. 表名以名词或名词短语单数命名
  5. 同一个功能或业务模块的表使用同一个前缀，例如 user、user_address、user_auth
  6. 关键字全部使用大写字母
  7. 表达是否概念的字段，使用 is_xxx 的方式命名，数据类型为 TINYINT(1) UNSIGNED，0 表示否，1 表示是，默认值为 0，例如 is_delete（是否被删除）
  8. 唯一索引使用 uk_xxx 命名，普通索引使用 index_xxx 命名

数据类型

  1. 尽量选用更小的数据类型
  2. 时间全部使用 INT(10) 类型存储，某些场景下如果需要存储到毫秒级则使用 INT(13)
  3. 尽量避免枚使用举类型，可使用 TINYINT 来代替，好处是好扩展
  4. 为每个非负字段添加 UNSIGNED，节约存储空间
  5. 小数使用 DECIMAL(m,n)，禁止使用 FLOAT 和 DOUBLE，因为后两种类型存在精度损失的问题，对于精度要求很高，或者超出 DECIMAL 范围的场景可以将整数和小数部分使用 INT 分开存储
  6. 如果存储的字符串长度几乎相等，使用 CHAR 定长字符串类型
  7. VARCHAR 是变长字符串，不预先分配存储空间，长度最好不要超过 5000，大于 5000 的字符串，使用 TEXT 存储，独立出来，使用主键关联，避免影响其他字段的索引

其他规则

  1. 每个表都必须创建一个自增 id，类型为 BIGINT UNSIGNED，并且设置为主键，理论上不参与到业务逻辑代码中
  2. 必须为每个表和字段设置合适的默认值和简单的注释
  3. 如无特殊用途，都使用 InnoDB 存储引擎
  4. 如果要存储表情则使用 utfmb4 字符集，否则都使用 utf-8

索引规则

  1. 具有唯一特性的字段，一定要加唯一索引
